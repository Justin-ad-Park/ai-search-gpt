## 2. 불용어 적용 방식 제안

### 2.1 운영 active 파일
- 새 파일:
  - `.local-nas/userdict/stopword.txt`
- 비교 검증용 샘플 파일:
  - `.local-nas/userdict/stopword.with-stopword.txt`
  - `.local-nas/userdict/stopword.without-stopword.txt`
- 초기 내용:

```txt
식감
```

설명:
- `user_dict_ko.txt`와 같은 디렉터리에 둔다.
- ES Pod 내부 경로:
  - `/usr/share/elasticsearch/config/userdict/stopword.txt`
- `stopword.txt`를 active 파일로 두고, `with/without` 샘플을 복사해 비교 검증한다.

### 2.2 ES 매핑 반영 방식
`index-mapping.json`의 `analysis.filter`에 stop filter를 추가한다.

예시:

```json
"ko_stop_filter": {
  "type": "stop",
  "stopwords_path": "userdict/stopword.txt"
}
```

그리고 analyzer filter 체인에 연결한다.

예시:

```json
"ko_mall_analyzer": {
  "type": "custom",
  "tokenizer": "ko_nori_userdict_tokenizer",
  "filter": ["lowercase", "ko_stop_filter", "ko_pos_filter"]
},
"ko_mall_search_analyzer": {
  "type": "custom",
  "tokenizer": "ko_nori_userdict_tokenizer",
  "filter": ["lowercase", "ko_stop_filter", "ko_synonym_filter", "ko_pos_filter"]
}
```

이번 구현 기준 filter 순서:
- `lowercase -> ko_stop_filter -> ko_synonym_filter -> ko_pos_filter`

### 2.3 vector 임베딩 처리 원칙
- `goods_name`, `goods_full_name`, `search_keyword` 등의 ES text 필드는 stop filter 적용
- 문서 벡터 생성용 텍스트는 원문 그대로 사용
- 검색어(query) 벡터 생성용 텍스트도 원문 그대로 사용

한 줄 요약:
- lexical은 stopword 제거
- vector는 원문 유지

## 6. 권장 검증 방식

### 6.1 `_analyze` 검증
이번 작업의 핵심 검증 기준은 검색 결과가 아니라 `_analyze` 결과다.

비교 검증 방식:
1. `stopword.with-stopword.txt`를 `stopword.txt`로 복사
2. 롤아웃 실행
3. `_analyze` 결과에서 `식감`이 제거되는지 확인
4. `stopword.without-stopword.txt`를 `stopword.txt`로 복사
5. 다시 롤아웃 실행
6. `_analyze` 결과에서 `식감`이 남아 있는지 확인

예시 입력:
- `바삭한 식감 만두`
- `식감 좋은 만두`

판정 기준:
- with-stopword: `식감` 제거
- without-stopword: `식감` 유지
- `만두` 같은 핵심 토큰은 남아 있어야 한다.

### 6.2 vector 영향 검증
확인 포인트:
- query embedding 입력에서 `식감`이 제거되지 않는지
- 문서 embedding 생성용 텍스트도 stopword 전처리를 타지 않는지

## 7. 확정된 의사결정

### 7.1 stopword를 index/search analyzer 모두에 적용할지
- 둘 다 적용

이유:
- 운영 절차가 단순하다.
- userdict와 유사하게 “수정 -> 롤아웃 -> 검증” 흐름으로 맞출 수 있다.
- 색인/검색 analyzer 기준이 같아져 디버깅이 쉽다.

### 7.2 stopword 파일 위치를 userdict 디렉터리에 같이 둘지
- 같이 둔다.

이유:
- 현재 PV/PVC와 mount 구성을 그대로 재사용 가능
- 운영자가 active 파일을 한 위치에서 관리 가능

### 7.3 vector 임베딩에 stopword를 적용할지
- 적용하지 않는다.

이유:
- vector 검색은 원문 문맥을 최대한 유지하는 편이 자연스럽다.
- lexical stopword와 vector 의미 표현의 역할을 분리할 수 있다.

### 7.4 불용어 검증 스크립트를 userdict 스크립트와 통합할지
- 분리

이유:
- tokenizer 검증과 stop filter 검증은 목적이 다르다.
- 장애 원인 추적도 분리된 편이 쉽다.

## 9. 위험 요소

1. 불용어를 너무 많이 넣으면 검색 recall이 과도하게 줄 수 있다.
2. `식감`처럼 설명성 단어라도 실제 상품 검색에서 의미가 있을 수 있다.
3. stop filter 적용 순서가 잘못되면 synonym/형태소 분석 결과가 예상과 달라질 수 있다.
4. index/search analyzer 둘 다 바꾸면 반드시 롤아웃과 회귀 검증이 필요하다.
5. lexical과 vector가 서로 다른 전처리 기준을 가지므로, 하이브리드 해석 혼동에 주의한다.

권장 대응:
- 초기는 `식감` 1개만 넣고 작게 시작
- `_analyze`와 실제 검색 둘 다 확인
- 문제 시 stopword 파일만 롤백하고 재롤아웃
